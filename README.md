# Парсер отзывов Ozon

PHP-парсер, который собирает отзывы о товарах с маркетплейса Ozon. PHP 8.4, Swoole, Symfony, PostgreSQL, Redis.

---

## Исследование API Ozon

### Как были найдены запросы

Для поиска API-запросов использовался **Chrome DevTools → вкладка Network**:

1. Открыл страницу товара на Ozon (например, `https://www.ozon.ru/product/...`).
2. Перешёл на вкладку «Отзывы» и скопировал фрагмент текста отзыва со страницы.
3. В DevTools Network использовал функцию **Search** (Ctrl+F в панели Network) — вставил скопированный текст отзыва и выполнил поиск по телам ответов всех запросов.
4. Нашёл запрос, в теле ответа которого содержится текст отзыва — это оказался единый API-endpoint Ozon.
5. Проанализировал структуру запроса и ответа, нашёл все поля отзывов и механизм пагинации.

Этот же подход применялся для поиска данных о товарах и категориях — копируем значение со страницы, ищем по Network, находим запрос.

### Найденные API-запросы

Ozon использует **единый JSON API** для всех данных на страницах:

```
GET https://www.ozon.ru/api/entrypoint-api.bx/page/json/v2?url=<внутренний_url>
```

Параметр `url` содержит внутренний путь страницы с параметрами пагинации. Для отзывов:

**Первая страница отзывов:**
```
url=/product/<slug>-<id>/?layout_container=reviewshelfpaginator&layout_page_index=3&tab=reviews&sort=published_at_desc
```

**Последующие страницы:**
Из ответа первой страницы берётся поле `nextPage` — это готовый URL для следующего запроса, который передаётся в тот же `url=` параметр.

### Структура ответа

Ответ API — JSON с ключом `widgetStates`, внутри которого виджеты в виде JSON-строк. Для отзывов используется виджет `webListReviews`:

```json
{
  "widgetStates": {
    "webListReviews-...": "{\"reviews\": [{...}, {...}]}",
    ...
  },
  "nextPage": "/product/...?layout_container=reviewshelfpaginator&layout_page_index=4&..."
}
```

Каждый виджет — JSON-строка, которую нужно дополнительно декодировать (`json_decode`). Внутри `webListReviews` — массив `reviews[]` с полными данными отзывов.

### Собираемые поля отзывов

| Поле из ТЗ | Путь в JSON ответа | Описание |
|---|---|---|
| Текст отзыва | `reviews[].content.comment` | Основной текст отзыва |
| Достоинства | `reviews[].content.positive` | Плюсы товара |
| Недостатки | `reviews[].content.negative` | Минусы товара |
| Дата публикации | `reviews[].publishedAt` | Unix timestamp или ISO-строка |
| Картинка | `reviews[].content.photos[].url` | Массив URL фотографий |
| Текст первого ответа | `reviews[].comments.list[0].text` | Первый ответ (обычно от продавца) |

Дополнительно собираются: автор (`author.firstName` + `author.lastName`), рейтинг (`content.score`), UUID отзыва (`uuid`).

Все поля nullable — если поле отсутствует в ответе, оно сохраняется как `null`.

## Трудности

### 1. Анти-бот защита Ozon

Главная трудность — Ozon активно блокирует автоматические запросы. После нескольких запросов с одного IP сервер начинает отвечать HTTP 403. Простой HTTP-клиент (Guzzle/cURL) без cookies и корректных заголовков блокируется почти сразу.

**Решение:** Для получения валидной сессии (cookies, User-Agent, Client Hints) используется внешний solver-service на базе Playwright (headless-браузер). Solver открывает страницу Ozon в реальном браузере, проходит проверки защиты и возвращает cookies. Дальнейшие API-запросы выполняются обычным HTTP-клиентом (Guzzle) с этими cookies.

### 2. Два формата виджетов отзывов

В ходе разработки обнаружилось, что Ozon использует два формата для отзывов:
- **`webListReviews`** — текущий формат, один виджет содержит массив `reviews[]` со всеми отзывами на странице.
- **`webReviewItem`** — устаревший формат, каждый отзыв приходит в отдельном виджете.

Парсер поддерживает оба формата с приоритетом `webListReviews` и фолбеком на `webReviewItem`.

### 3. Пагинация через `nextPage`

Ozon не использует классическую пагинацию `?page=N`. Параметры пагинации (`layout_container`, `layout_page_index`) должны быть вложены именно в параметр `url=`, а не в query string самого API-запроса — иначе сервер их игнорирует и возвращает пустой ответ. Это было выяснено эмпирически, т.к. API не документирован.

Следующая страница определяется полем `nextPage` в ответе. Когда отзывы заканчиваются — поле пустое или отсутствует.

### 4. Необходимость прокси

При парсинге 100 товаров подряд один IP быстро попадает в бан. Для стабильной работы необходима ротация прокси-серверов. В парсере реализован Identity Pool — каждый прокси получает свою сессию (cookies + UA), и при блокировке одного прокси система переключается на другой.

### 5. Виджеты приходят как JSON-строки

Значения в `widgetStates` — не JSON-объекты, а **строки**, которые нужно отдельно декодировать через `json_decode`. Это не очевидно при первом анализе ответа и требует дополнительного шага парсинга.

## Как запустить парсинг 100 товаров

### Вариант 1: Загрузка через админ-панель (рекомендуется)

1. Подготовить файл со списком товаров. Поддерживаемые форматы:

   **TXT** — один ID или URL на строку:
   ```
   1234567
   9876543
   https://www.ozon.ru/product/nazvanie-tovara-1234567/
   ```

   **CSV** — один столбец:
   ```csv
   1234567
   9876543
   ```

   **JSON** — массив ID:
   ```json
   [1234567, 9876543, 1111111]
   ```

2. Открыть админ-панель: `http://localhost:8202/tasks/create`
3. Выбрать тип задачи: **Парсинг отзывов**
4. Загрузить файл со 100 ID товаров
5. Нажать «Создать задачу»

Система создаст 100 задач (по одной на товар) и поместит их в Redis-очередь. Swoole-воркер автоматически обработает их последовательно.

### Вариант 2: Парсинг товаров с автоматическим сбором отзывов

Если нужно параллельно собрать информацию о товаре (цена, название, характеристики) и его отзывы:

1. При создании задачи выбрать тип **Парсинг отзывов** и отметить **«Собрать данные товара»**
2. Система создаст для каждого товара дочернюю задачу `product` (сбор карточки), после завершения которой автоматически запустится задача `reviews` (сбор отзывов)

### Мониторинг

- **Список задач:** `http://localhost:8202/tasks/` — статусы, прогресс, ошибки
- **Логи парсера:** `make logs-parser` — детальные логи каждого запроса
- **Собранные товары:** `http://localhost:8202/products/`
- **Здоровье системы:** `http://localhost:8202/health/` — статус всех сервисов и Identity Pool

### Настройка глубины сбора отзывов

По умолчанию парсер собирает отзывы за **последний месяц** (параметр `date_from`). Это поведение задано в `ReviewsTaskHandler`:
- Пагинация останавливается, когда все отзывы на странице старше `date_from`
- Максимальное число страниц: `MP__PARSER__MAX_REVIEW_PAGES` (по умолчанию 10)

## Solver-service (внешняя зависимость)

### Зачем нужен

Ozon использует анти-бот защиту, которая блокирует запросы без валидных browser cookies. Solver-service — это отдельный сервис на базе **Python + Playwright** (headless-браузер), который:
1. Открывает страницу Ozon в реальном Chromium
2. Проходит проверки защиты (JavaScript challenges, fingerprinting)
3. Возвращает полученные cookies, User-Agent и Client Hints

Парсер использует эти данные для обычных HTTP-запросов через Guzzle — это быстрее и надёжнее, чем загружать каждую страницу через браузер.

### Как работает интеграция

```
[Parser] --Guzzle--> [Ozon API]     ← быстрый путь (с cookies от solver)
    |                     |
    |  403 Forbidden      |
    v                     |
[Parser] --HTTP--> [Solver] --Playwright--> [Ozon]
                       |
                   cookies + UA
```

1. Parser пробует запрос через Guzzle с имеющимися cookies
2. Если получает 403 — отправляет URL в solver для browser fetch
3. Solver возвращает данные + обновлённые cookies
4. Cookies кешируются в Identity Pool для последующих запросов

### Настройки подключения

```env
MP__SOLVER__HOST=host.docker.internal   # Адрес solver-service
MP__SOLVER__PORT=8204                   # Порт
MP__SOLVER__REQUEST_TIMEOUT=90          # Таймаут запроса (секунды)
MP__SOLVER__CONNECT_TIMEOUT=10          # Таймаут подключения
```

> **Примечание:** Solver-service поставляется отдельно. Без solver парсер может работать с прокси, которые ещё не заблокированы Ozon, но стабильность не гарантирована.

## Архитектурные решения

### Почему Swoole

Парсинг — IO-bound задача (сетевые запросы, ожидание ответа). Swoole с корутинами позволяет:
- Обрабатывать задачи асинхронно без блокировки event loop
- Использовать пулы соединений к PostgreSQL и Redis (не пересоздавать соединение на каждый запрос)
- Устанавливать таймауты на уровне корутин — зависшая задача не блокирует воркер

### Почему Redis-очереди

Задачи парсинга ставятся в Redis-очередь (LPUSH/BRPOP). Это даёт:
- Персистентность — задачи не теряются при перезапуске парсера
- Дедупликация — один и тот же товар не парсится дважды в рамках одной задачи
- Прогресс-трекинг — текущее состояние задачи доступно в реальном времени
- Распределённые блокировки — предотвращают одновременную обработку одной задачи

### Почему Identity Pool

Каждая «identity» — это связка прокси + cookies + User-Agent. Пул решает:
- **Ротация при блокировке:** при 403 текущая identity уходит в карантин, захватывается новая
- **Переиспользование сессий:** не нужно вызывать solver на каждый запрос
- **Учёт TTL:** cookies протухают — identity автоматически удаляется и создаётся заново
- **Прогрев:** Identity Warmer заранее создаёт identity для каждого прокси перед началом парсинга

## Прокси

Для стабильной работы при парсинге большого числа товаров рекомендуется использовать прокси-серверы.

```env
MP__PROXY__ENABLED=true
MP__PROXY__LIST=http://user:pass@proxy1:8080,http://user:pass@proxy2:8080
```

Поддерживаемые протоколы: `http`, `https`, `socks5`. Прокси ротируются через Identity Pool — каждый прокси получает свою сессию.

## Трассировка (trace_id)

Каждая задача помечается уникальным `trace_id` (UUID v4):
- Передаётся через заголовок `X-Trace-Id` между парсером и solver-service
- Сохраняется в логах и в БД (`parse_logs`)
- Позволяет отследить весь путь обработки: от создания задачи до сохранения отзывов
